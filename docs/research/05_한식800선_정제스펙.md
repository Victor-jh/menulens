## 2. Supabase Postgres 스키마

### 2.1 테이블 정의

```sql
-- pgvector 확장 활성화 (D3 최초 1회)
CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE TABLE hansik_800 (
    id              SERIAL PRIMARY KEY,
    source_no       INTEGER NOT NULL UNIQUE,              -- 원본 요리번호 1~800
    category        TEXT NOT NULL,                        -- "찌개 [Jjigae]", "구이 [Gui]" (단일 23개 카테고리)
    name_ko         TEXT NOT NULL,                        -- 김치찌개
    romanization    TEXT NOT NULL,                        -- Kimchijjigae (공식 라틴 발음)
    name_en         TEXT NOT NULL,                        -- Kimchi Stew
    name_ja         TEXT,                                 -- キムチチゲ
    name_zh_simp    TEXT,                                 -- 辛奇汤
    name_zh_trad    TEXT,                                 -- 辛奇湯
    desc_ko         TEXT,                                 -- 재료·조리법 설명 (재료 추출 소스)
    desc_en         TEXT,
    desc_ja         TEXT,
    desc_zh_simp    TEXT,
    desc_zh_trad    TEXT,

    -- §3 후처리 생성 (로더가 desc_ko·name_ko 파싱으로 추출)
    ingredients_extracted   TEXT[],                       -- ['돼지고기','두부','파','마늘'] (desc_ko 추출)
    allergy_tags    TEXT[],                               -- ['pork','soy','seafood'] (§3 매핑)
    vegan_status    TEXT CHECK (vegan_status IN ('vegan','vegetarian','non_veg','unknown')),
    halal_status    TEXT CHECK (halal_status IN ('halal_likely','haram','unclear')),
    spicy_level     INTEGER CHECK (spicy_level BETWEEN 0 AND 3),

    created_at      TIMESTAMPTZ DEFAULT NOW(),
    updated_at      TIMESTAMPTZ DEFAULT NOW(),

    -- 임베딩 (§4) — ADR-008 이후 차원·프로바이더 확정
    embedding_ko    VECTOR(1536)                          -- OpenAI: 1536 / Gemini text-embedding-004: 768 (ADR-008 선택 시 스키마 조정 필요)
);

-- 유사도 검색 인덱스 (HNSW)
CREATE INDEX hansik_800_embedding_ko_idx
    ON hansik_800 USING hnsw (embedding_ko vector_cosine_ops);

-- 텍스트 검색 인덱스 (정확 매칭 · fallback)
CREATE INDEX hansik_800_name_ko_trgm_idx
    ON hansik_800 USING gin (name_ko gin_trgm_ops);
```

**스키마 변경 사항 (2026-04-23 실 xlsx 반영)**:
- `category_major`/`category_minor` 삭제 → 단일 `category`
- `source_row_no` → `source_no` (원본 `요리번호` 필드명 정합)
- `romanization` 추가 (필수, NOT NULL)
- `name_en_desc` 제거 — 원본에 없음 (설명은 `desc_en`으로 분리)
- `main_ingredients` → `ingredients_extracted` (원본 필드 없음, 로더가 `desc_ko`에서 추출)
- `image_path` 제거 (원본에 없음)
- 다국어 설명 4개 (`desc_en`·`desc_ja`·`desc_zh_simp`·`desc_zh_trad`) 모두 컬럼 추가

### 2.2 인덱스 선택 근거
- **HNSW**: pgvector 0.5+ 기본, Recall@10 > 0.95, latency < 50ms (800행 규모)
- **trgm (trigram)**: 메뉴명 부분일치/오타 보정 (예: "김찌" → "김치찌개")
- **향후 확장 시**: 영·일·중 임베딩 추가 고려 — 지금은 한국어만 (Menu Reader가 한국어로 뽑아줌)

### 3.2 추출 파이프라인 (로더 책임)

원본에 `main_ingredients` 컬럼이 없으므로 로더(`backend/scripts/load_hansik_800.py`)가 다음 2단계로 재료·태그를 생성:

**Stage 1: 규칙 기반 키워드 매칭 (빠르고 재현성 있음)**
```python
def extract_tags(text: str) -> list[str]:
    # text = name_ko + ' ' + desc_ko 결합
    tags = []
    for tag, keywords in TAG_KEYWORDS.items():
        if any(kw in text for kw in keywords):
            tags.append(tag)
    return tags
```
김치찌개 예시 (desc_ko 발췌: "돼지고기나 어패류, 두부, 굵게 썬 파"):
- "돼지고기" → `pork`
- "어패류" → `seafood`
- "두부" → `soy`
- 매운맛 서술 "매콤한"·"얼큰한" → `spicy_level=2`

**Stage 2: LLM 보조 (애매·누락 케이스 — D5 작업, Phase 1 optional)**
Claude Sonnet에 `desc_ko` + 후보 태그 리스트 전달해 structured output으로 확정. Phase 1 비용 절감 위해 **Stage 1만 사용, Stage 2는 Phase 2로 연기** 가능 (규모 800건).

**규칙**:
- Stage 1 매칭만으로 충분 커버리지(김치찌개 3태그)라 판단 — 실 적재 후 샘플 20건 수동 검증 필수
- 복수 태그 허용
- 의심 케이스는 `halal_status='unclear'` 처리 (미림·맛술 등)

### 3.2 비건/베지테리언/할랄 자동 분류

**`vegan_status`**:
- `vegan`: 고기·해물·생선·계란·유제품 태그 모두 없음
- `vegetarian`: 고기·해물·생선 태그 없음 (계란·유제품은 허용)
- `non_veg`: 위 외 전부
- `unknown`: 재료 정보 부족

**`halal_status`**:
- `haram`: `pork`, `alcohol` 중 하나라도 포함
- `halal_likely`: 위 해당 없음 + 수동 확인 필요
- `unclear`: 미림·조미료 등 숨은 알코올 의심 재료 존재

**주의**: `halal_likely`는 **KMF 인증과 다름**. 제안서·UI에서 "할랄 인증 = KMF 목록 교차 확인"으로 분리 명시.

### 3.3 매운맛 강도 (`spicy_level`)

| 레벨 | 의미 | 자동 분류 키워드 |
|---:|---|---|
| 0 | 안 매움 | 기본 |
| 1 | 약간 매움 | 고춧가루, 고추장 소량 (범위 판단 어려움 → 수동) |
| 2 | 매움 | 김치, 김치찌개, 제육볶음 |
| 3 | 매우 매움 | 불닭, 낙곱새, 해물찜 매운맛, 매운탕 |

**한계**: 자동 규칙으로는 정확 분류 불가. Claude Sonnet에 `desc_ko` 전달 → structured output으로 분류 요청 (D5 작업).

## 4. 임베딩 생성 규칙

### 4.1 대상 텍스트 조합 (실 컬럼 반영)

`embedding_ko` 생성 시 아래 형식으로 한 문장 합성 후 임베딩:

```
{name_ko} {romanization} {name_en} {category_ko_only} {desc_ko_first_40chars}
예: 김치찌개 Kimchijjigae Kimchi Stew 찌개 신김치를 이용한 국물요리다 돼지고기 두부
```

**근거**:
- 메뉴명만으로는 희소 (800건, 어휘 다양성 부족)
- `romanization`·영문·카테고리 포함 → 다국어 검색 폴백
- `desc_ko` 앞 40자 포함 → 재료·조리법 맥락 임베딩 (ingredients_extracted 대신)
- 카테고리는 Latin 발음 부분 제거(`[Jjigae]` 제거 → `찌개`만)해 노이즈 축소

**차원·프로바이더**:
- 기본: OpenAI `text-embedding-3-small` (1536차원, $0.00002/1K tokens)
- 대안: Gemini `text-embedding-004` (768차원, 무료 티어) — ADR-008 결정 필요
- JH가 이미 `GEMINI_API_KEY` 보유 상태라 Gemini 대안이 비용·작업 효율 측면 우세

### 4.2 임베딩 예산
- 800건 × 평균 30토큰 = 24,000 토큰
- $0.00048 = **0.5원 미만** (1회 적재 비용)
- 재실행 방지: `updated_at` 기준 diff 감지 후 변경분만 재임베딩

### 4.3 유사도 검색 임계값
- Menu Reader가 뽑은 메뉴명 → pgvector 유사도 검색
- `cosine_distance < 0.35` 매칭 허용
- `0.35 <= distance < 0.50` → 후보 3개 제시 + Claude Sonnet 최종 선택
- `distance >= 0.50` → 매칭 실패, ⚪(정보없음) 처리
